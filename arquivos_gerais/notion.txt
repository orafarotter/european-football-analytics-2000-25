dbt CONHECIMENTO

Em projetos com dbt, o schema.yml é colocado dentro das pastas (staging, marts, etc.) por organização lógica, manutenção e governança dos modelos. A estrutura não é obrigatória, mas segue boas práticas amplamente adotadas.

Erro de PATH no dbt (Exit code 2): O Airflow não encontrava o executável do dbt no container. Solução: Parametrizamos o caminho absoluto (/home/airflow/.local/bin/dbt) diretamente na DAG.

Permissão negada na Service Account: A task upload_to_gcs não conseguia ler a chave JSON mapeada no Docker. Solução: Executamos chmod 644 no arquivo localmente (host) para liberar a leitura.

Falha de compilação por variáveis: O profiles.yml buscava uma variável de ambiente diferente da definida, quebrando o dbt antes de rodar. Solução: Fixamos o ID do projeto e o caminho da credencial diretamente no perfil.

Bloqueio de escrita de logs/target: O dbt não tinha permissão para gravar os arquivos compilados na pasta mapeada do host. Solução: Alteramos a DAG para copiar o projeto para a pasta /tmp (livre para escrita), rodar de lá e apagar depois.

Erro 400 no BigQuery (Location Mismatch): O dbt seed falhou pois tentava carregar dados na região global (US) contra um dataset criado na Carolina do Sul. Solução: Ajustamos o parâmetro location no profiles.yml estritamente para us-east1.






Por que view no staging e table no mart?
A regra geral no dbt é: materialização acompanha o propósito da camada
Staging é uma camada de transformação intermediária, não de consumo. Ela existe apenas para limpar, tipar e padronizar os dados brutos — ninguém consome o staging diretamente, nem o Looker Studio, nem analistas. Ela é um "degrau" para chegar no mart.
Como é apenas um degrau, não faz sentido gastar dinheiro e tempo materializando ela em disco. A view é só uma query salva — ela não ocupa espaço no BigQuery e não tem custo de escrita. Quando o modelo fct_european_matches referencia stg_matches, o BigQuery simplesmente expande a view inline e executa tudo em uma única query.
Mart é a camada de consumo final. O Looker Studio vai bater nessa tabela potencialmente várias vezes por dia, com filtros, agregações e múltiplos usuários simultâneos. Se fosse uma view, o BigQuery teria que reprocessar os 76.7k rows e 41.8 MiB a cada clique no dashboard. Como é uma table particionada e clusterizada, os dados já estão em disco, organizados — a query é muito mais rápida e barata.
