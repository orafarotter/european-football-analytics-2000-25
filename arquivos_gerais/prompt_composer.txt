## 🎯 Objetivo do Projeto

Construir um **pipeline de dados end-to-end** (batch) usando o dataset público de partidas de futebol do Kaggle ([Club Football Match Data 2000–2025](https://www.kaggle.com/datasets/adamgbor/club-football-match-data-2000-2025)), com foco nas **principais ligas europeias**, para alimentar um dashboard analítico no **Looker Studio**.

## 📦 Sobre o Dataset

### Fonte
- **URL Kaggle**: https://www.kaggle.com/datasets/adamgbor/club-football-match-data-2000-2025
- **Arquivo**: CSV único com partidas de futebol de 2000 a 2025 de diversas ligas ao redor do mundo.

### De-para de ligas (será usado como seed no dbt — não como arquivo externo)
O arquivo `ligascsv.xlsx` contém o mapeamento entre o código da liga e seu nome/país. Ele deve ser transformado em uma **dbt seed** (arquivo CSV dentro do projeto dbt), para que o join com os dados brutos seja feito no próprio dbt Cloud, sem dependência de arquivos externos.

```
Code  | Country        | League
------|----------------|----------------------------
E0    | England        | Premier League
SP1   | Spain          | La Liga
D1    | Germany        | Bundesliga
I1    | Italy          | Serie A
F1    | France         | Ligue 1
N1    | Netherlands    | Eredivisie
P1    | Portugal       | Primeira Liga
T1    | Turkey         | Süper Lig
SC0   | Scotland       | Premiership
B1    | Belgium        | Pro League
... (lista completa no seed)
```

> **Nota**: O seed completo deve conter todas as 38 ligas do arquivo original.

## 🗺️ Arquitetura do Pipeline

```
[Kaggle URL]
     │
     ▼ (Python via Composer DAG)
[Cloud Storage — raw/ ]   ← política: sobrescrever se já existir
     │
     ▼ (BigQuery External Table via Composer DAG)
[BigQuery — dataset: eu_football_raw]
     │
     ▼ (dbt Cloud — triggered via Composer DAG ou API)
[BigQuery — dataset: eu_football_staging]   ← modelos staging (stg_)
     │
     ▼
[BigQuery — dataset: eu_football_mart]      ← modelos mart particionados e clusterizados
     │
     ▼
[Looker Studio — Dashboard]
```

### Modelo "Medallion"
- **Bronze/Raw**: dado bruto no GCS + tabela externa no BigQuery (`eu_football_raw`)
- **Silver/Staging**: limpeza, cast de tipos, join com seed de ligas (`eu_football_staging`)
- **Gold/Mart**: tabela final agregada, particionada por ano e clusterizada por liga, filtrada pelas principais ligas europeias (`eu_football_mart`)

## 📋 Etapas do Projeto

### Etapa 0 — Terraform: Infraestrutura e Permissões no GCP
Criar todos os recursos GCP via Terraform na região `us-east1`:

- **Service Account** dedicada ao projeto com as permissões mínimas necessárias (principle of least privilege):
  - `roles/storage.admin`
  - `roles/bigquery.admin`
  - `roles/composer.worker`
- **Cloud Storage Bucket** para o data lake (`eu-football-raw-20-25`)
- **BigQuery Datasets**: `eu_football_raw`, `eu_football_staging`, `eu_football_mart`
- **Cloud Composer Environment** (Airflow)
- Outputs do Terraform: bucket name, dataset IDs, service account email

> 💡 Dica do sênior: use `terraform.tfvars` para variáveis sensíveis e nunca commite chaves no repositório. Use `.gitignore` adequado.

### Etapa 1 — Extração: Download do CSV via Python
Script Python que:
- Faz o download do dataset a partir de um link direto (Kaggle API ou URL pública)
- Salva localmente como CSV antes de enviar ao GCS

---

### Etapa 2 — Carga no Data Lake: Upload para Cloud Storage
- Upload do CSV para o bucket GCS no path `raw/matches.csv`
- **Política de sobrescrita**: se o arquivo já existir, sobrescrever (usar `if_generation_match=0` ou equivalente no SDK do GCS)
- O arquivo deve ser carregado de forma idempotente

---

### Etapa 3 — Tabela Externa no BigQuery
- Criar uma tabela externa no dataset `eu_football_raw` apontando para o arquivo CSV no GCS
- Carregar **todas as colunas** do CSV na tabela externa (sem filtro de colunas)
- Schema explícito (não auto-detect) para garantir tipos corretos
- A seleção das colunas relevantes para o projeto acontece no modelo staging do dbt

---

### Etapa 4 — Orquestração com Cloud Composer (Airflow)
Criar uma DAG com as seguintes tasks encadeadas:

```
download_csv >> upload_to_gcs >> create_external_table >> trigger_dbt_job
```

- `download_csv`: executa o script Python de extração
- `upload_to_gcs`: faz upload com política de sobrescrita
- `create_external_table`: cria ou recria a tabela externa no BigQuery
- `trigger_dbt_job`: chama a API do dbt Cloud para rodar o job de transformação

> 💡 Boas práticas: Use `GCSHook` e `BigQueryHook` nativos do Airflow. Defina `retries`, `retry_delay` e `on_failure_callback`. Use variáveis do Airflow (não hardcode credenciais).

---

### Etapa 5 — Transformações com dbt Cloud

#### Estrutura do projeto dbt:
```
football_dbt/
├── models/
│   ├── staging/
│   │   └── stg_matches.sql          ← limpeza + cast + join com seed de ligas
│   └── mart/
│       └── fct_european_matches.sql ← filtro das ligas europeias + agregações
├── seeds/
│   └── leagues.csv                  ← de-para de ligas (conteúdo do ligascsv.xlsx)
├── tests/
│   └── ...                          ← testes de qualidade de dados
├── macros/
│   └── ...
└── dbt_project.yml
```

#### Modelo `stg_matches`:
- Cast de tipos (`MatchDate` como DATE, gols como INTEGER, etc.)
- Join com `ref('leagues')` para trazer `country` e `league_name`
- Renomear colunas para snake_case padronizado
- Filtrar registros com `MatchDate IS NOT NULL`

#### Modelo `fct_european_matches`:
- Filtrar pelas principais ligas europeias via `WHERE division IN ('E0', 'SP1', 'I1', 'D1', 'F1', 'B1', 'P1', 'E1', 'DEN', 'POL')`
- **Particionamento**: por coluna `match_year` (extraída do `MatchDate`) — `PARTITION BY DATE_TRUNC(match_date, YEAR)`
- **Clustering**: por `division` e `league_name`
- Colunas calculadas: `total_goals`, `goal_difference`, `match_result_label`, etc.

#### Seeds:
- `leagues.csv` com todo o conteúdo do `ligascsv.xlsx` (38 ligas)

#### Testes dbt (para "going the extra mile"):
- `not_null` nos campos críticos
- `accepted_values` para `FTResult` (H, A, D)
- `relationships` entre modelos
